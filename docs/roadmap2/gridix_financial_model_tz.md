# Техническое задание (ТЗ)
## Финансовая модель проекта GRIDIX

---

## 1. Цель документа

Данное ТЗ предназначено для финансового аналитика / CFO / финансового моделиста, который будет **с нуля или на базе черновика** собирать профессиональную финансовую модель проекта **GRIDIX**.

Модель должна быть:
- понятна **классическому венчурному инвестору / фонду**,
- масштабируемая (5 лет / 60 месяцев),
- прозрачная по логике и формулам,
- легко редактируемая без риска сломать расчёты,
- пригодная для отправки в фонды и для использования в pitch deck.

Формат: **Google Sheets или Excel**.

---

## 2. Кратко о продукте GRIDIX (контекст для аналитика)

GRIDIX — SaaS‑платформа для рынка недвижимости.

Целевые сегменты:
1. **Девелоперы** (B2B, основной чек)
2. **Агентства недвижимости** (B2B, средний чек)
3. **Агенты** (массовый сегмент, low‑ticket)

Монетизация — подписка (MRR).
Рост — через маркетинговую воронку + партнёрства.

---

## 3. Горизонт и сценарии

### 3.1. Горизонт модели

- **5 лет / 60 месяцев**
- Помесячная детализация

### 3.2. Сценарии

Обязательно реализовать:
- **Base**
- **Optimistic**
- **Pessimistic**

Сценарии должны влиять через мультипликаторы:
- воронка / конверсии,
- цены (MRR),
- расходы.

Переключение сценария — одной ячейкой.

---

## 4. Структура листов (обязательная)

Финальная модель должна содержать следующие листы:

1. `ReadMe`
2. `Assumptions`
3. `Scenarios`
4. `Headcount`
5. `Funnel`
6. `Customers`
7. `P&L`
8. `Summary`
9. `UnitEconomics`
10. `Dashboard`

---

## 5. ReadMe (инструкция)

На листе `ReadMe` должно быть текстовое описание:
- что делает каждый лист,
- какие ячейки **можно менять**,
- какие ячейки **трогать нельзя** (формулы),
- пример: «как добавить сотрудника и увидеть влияние в EBITDA».

Язык: **простой, для нефинансового фаундера**.

---

## 6. Assumptions (исходные допущения)

Лист для ручного редактирования.

Обязательные блоки:

### 6.1. Продукт и цены
- Средний MRR с девелопера
- Средний MRR с агентства
- Средний MRR с агента

### 6.2. Стартовая база клиентов
- Девелоперы
- Агентства
- Агенты

### 6.3. Churn (помесячный)
- Dev churn
- Agency churn
- Agent churn

### 6.4. Маркетинг
- Бюджет маркетинга (месяц 1)
- Рост бюджета MoM
- CPL
- Конверсии в платящих клиентов по сегментам

### 6.5. Расходы
- Прочие фиксированные расходы
- ФОТ (из Headcount)
- Переменные расходы (% от выручки)

### 6.6. Налоги и кэш
- Ставка налога на прибыль
- Стартовый кэш

## 7. Scenarios

Таблица сценариев:

| Scenario | Funnel Mult | Price Mult | Cost Mult |
|--------|------------|------------|-----------|
| Base | 1.0 | 1.0 | 1.0 |
| Optimistic | >1 | >1 | ~1 |
| Pessimistic | <1 | <1 | >1 |

Сценарии должны применяться через формулы (`INDEX + MATCH` или аналоги).

---

## 8. Headcount (команда)

Лист для моделирования команды.

Колонки:
- Отдел
- Роль
- Годовая зарплата (Year 1 / Year 2 / Year 3)

Требования:
- Внизу строка **"Итого по зарплатам"**
- ФОТ автоматически попадает в `Assumptions` (деление на 12)
- Добавление сотрудника **не ломает модель**

---

## 9. Funnel (маркетинговая воронка)

Помесячная таблица (60 строк):

Колонки:
- Месяц
- Маркетинговый бюджет
- Лиды
- Новые девелоперы
- Новые агентства
- Новые агенты

Формулы:
- Лиды = Бюджет / CPL
- Клиенты = Лиды × конверсия × сценарный множитель

---

## 10. Customers (активная база)

Расчёт активных клиентов по месяцам:

Формула логики:

Активные(t) = Активные(t‑1) × (1 − churn) + Новые(t)

Отдельно по:
- девелоперам
- агентствам
- агентам

+ строка «Всего клиентов».

---

## 11. P&L (основной лист)

Обязательные строки:

- MRR
- ARR
- Выручка месяца
- Фиксированные расходы
- Переменные расходы
- Итого расходы
- **EBITDA**
- **Налог на прибыль** (только при EBITDA > 0)
- Чистая прибыль месяца
- **Кэш на конец месяца**

Кэш считается накопительно от стартового.

---

## 12. Unit Economics

Обязательно:

- CAC по сегментам
- LTV по сегментам
- Срок окупаемости (Payback)
- LTV / CAC
- Blended ARPU (на 12‑м месяце)

Без ошибок формул (`#NAME?`, `#DIV/0!`).

## 17. Контакт и контекст

Модель делается для фаундера GRIDIX.
Фаундер — нефинансовый специалист, поэтому **логика важнее сложности**.

Главный критерий успеха:
> «Инвестор открывает файл и за 3–5 минут понимает бизнес, рост и экономику проекта».
